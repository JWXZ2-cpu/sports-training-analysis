# 脉 MAI — 产品设计文档

> 文档版本：v1.1
> 最后更新：2026-06-29
> 产品名称：脉 / MAI
> 产品定位：面向专业运动队的 AI 训练分析平台

---

## 一、为什么做这个产品

### 1.1 问题诊断

在与多支专业运动队（田径中长跑为主）的教练团队深度接触后，我们发现训练管理存在五个核心痛点：

#### 痛点一：训练反馈丢失在"口头对话"中

运动员每天完成训练后，教练会口头询问"今天感觉怎么样"。运动员回答"还行"、"最后两组有点掉速"、"膝盖有点酸"。这些信息对训练决策至关重要，但**从未被记录**。

教练回忆上周三某位运动员说"膝盖有点酸"时，已经记不清具体细节。更严重的是，当运动员连续三天提到"膝盖酸"，没有任何系统能自动识别这个趋势并发出预警。

**本质问题：训练反馈是口语化的、即时的、非结构化的，而传统训练管理系统要求结构化数据输入，两者之间存在巨大的录入鸿沟。**

#### 痛点二：教练与运动员之间存在"信息不对称"

教练需要知道运动员的真实身体和心理状态来调整训练计划，但运动员往往不愿意完整表达。原因包括：
- 担心说"太累"会被减训练量，影响成绩进步
- 不确定自己的感受是否"值得说"
- 面对教练时有心理压力，倾向于说"还行"

同时，教练也面临一个两难：如果让运动员看到完整的专业分析（ACWR 负荷比、VDOT 趋势、心理状态评估），运动员可能会：
- 过度关注数据而非感受
- 产生"AI 比教练还懂"的信任危机
- 根据数据自行调整训练，打乱教练计划

**本质问题：运动员需要被关心，教练需要真相，但现有的系统要么不关心运动员的感受，要么把所有专业数据一股脑展示给所有人。**

#### 痛点三：医训协同靠"口头传话"

队医在下午对运动员做了膝盖治疗，主教练不知道，第二天安排了间歇跑。运动员带伤训练，伤病加重。

这不是个例。在大多数运动队中，队医和教练之间的信息传递依赖：
- 运动员自己转述（"队医说我膝盖没事"——但队医原话可能是"暂时没事，但要避免高强度冲击"）
- 非正式的走廊对话
- 微信群里的零散消息

**本质问题：治疗决策和训练决策在物理上分离，但影响的身体部位高度重叠，缺乏系统化的冲突检测机制。**

#### 痛点四：外教面临语言障碍

越来越多的运动队聘请外教执教。外教需要：
- 看懂运动员的训练反馈（通常中文）
- 理解 AI 分析报告
- 与中文环境的管理系统交互

现有的训练管理系统几乎全部是纯中文界面，外教需要依赖翻译或助教才能使用。

**本质问题：国际化不是"翻译菜单"那么简单，而是要让外教在不懂中文的情况下独立完成训练管理工作。**

#### 痛点五：训练数据分散在多个工具中

运动员用手表记录配速和心率（Garmin/华为），用纸笔记训练感受，用微信给教练发语音，教练用 Excel 做训练计划，队医用另一个系统记伤病。数据碎片化导致：
- 无法形成运动员的完整画像
- 无法进行跨维度的关联分析（如：连续高负荷 + 情绪低落 + 部位疼痛 = 伤病风险）
- 教练做决策时信息不完整

**本质问题：数据孤岛导致决策盲区。**

### 1.2 目标用户画像

#### 主要用户：主教练

| 维度 | 描述 |
|---|---|
| **角色定位** | 训练计划的制定者和最终决策者 |
| **日常工作** | 制定周训练计划、监督执行、根据运动员状态调整计划、赛前减量安排 |
| **核心诉求** | 快速了解全队状态，不遗漏任何风险信号，高效制定科学的训练计划 |
| **痛点** | 信息碎片化、依赖记忆和经验做判断、缺乏数据支撑的周期化管理 |
| **技术水平** | 中等，能使用手机 App，不愿意学习复杂系统 |
| **使用场景** | 训练前查看计划、训练后查看反馈、晚间制定次日计划、赛前周期调整 |

#### 关键用户：运动员

| 维度 | 描述 |
|---|---|
| **角色定位** | 训练的执行者和反馈的来源 |
| **日常工作** | 按计划完成训练、记录训练感受、上传手表数据 |
| **核心诉求** | 被关心、被理解，不想被过多专业术语困扰 |
| **痛点** | 不知道该说什么、不想被打扰、担心说错影响训练安排 |
| **技术水平** | 参差不齐，但都用智能手机 |
| **使用场景** | 训练结束后 5 分钟内（最真实的感受窗口期） |

#### 协同用户：助教、队医、管理人员

| 用户 | 核心诉求 |
|---|---|
| **助教** | 主教练不在时记录训练情况，作为信息中转站 |
| **队医** | 管理伤病治疗记录，确保治疗不与训练冲突 |
| **管理人员** | 掌握出勤率和训练完成率，向领导汇报 |

---

### 1.3 设计原则

基于以上分析，我们确立了四条产品设计原则：

| 原则 | 含义 | 反例 |
|---|---|---|
| **降低输入门槛** | 让运动员愿意说、方便说，比让系统功能强大更重要 | 一个功能齐全但没人用的系统，不如一个功能简单但每天都在用的系统 |
| **信息分级透明** | 不同角色看到不同层次的信息，而非一刀切 | 把 ACWR 和 VDOT 直接展示给运动员，可能适得其反 |
| **AI 辅助而非替代** | AI 提供分析和建议，决策权始终在人 | "AI 说你今天应该休息"——这会削弱教练的权威 |
| **冲突前置检测** | 在问题发生前发现它，而非事后复盘 | 训练导致伤病加重后才知道队医昨天做过治疗 |

## 二、核心功能设计逻辑

### 2.1 为什么是"语音优先"而不是"表单优先"

#### 传统方案：表单录入

```
运动员打开 App → 填写训练日期 → 选择训练类型 → 输入距离 → 输入配速
→ 输入心率 → 评分身体状态(1-10) → 评分心理状态(1-10) → 输入训练感受文字
→ 提交
```

**问题**：
- 训练后运动员最不想做的事就是填表
- 9 个字段需要逐一填写，耗时 3-5 分钟
- 训练后 5 分钟是"感受黄金窗口"，过了这个时间记忆会衰减
- 表单设计再好也是"要求运动员适应系统"

#### 我们的方案：语音优先

```
运动员训练结束 → 打开 App → 按住麦克风说"今天跑的还行，最后两组有点掉速，
膝盖有点酸，心理状态挺好的" → 系统自动转录 → AI 自动解析出训练类型、身体状态、
心理状态、风险信号 → 运动员只需确认 → 提交
```

**优势**：
- 输入门槛极低：按住说话，30 秒完成
- 自然语言保留了丰富的情感和细节（"有点酸" vs "很痛" vs "隐隐约约的"）
- AI 解析后自动补全结构化数据，运动员无需理解什么是"ACWR"
- 符合训练后的真实行为模式：运动员训练后第一反应是"跟人说说今天练得怎么样"

#### 设计决策

| 决策 | 选择 | 理由 |
|---|---|---|
| 主输入方式 | 语音 | 降低门槛，抓住黄金窗口期 |
| 备选输入方式 | 文字 | 安静环境或运动员偏好文字时使用 |
| 辅助输入 | 手表 FIT 文件 | 自动采集客观数据（配速/心率/距离），无需手动输入 |
| 评分方式 | 滑块（1-10） | 仅需 3 次滑动，比填数字更快 |

### 2.2 为什么是"双视角"而不是"单视角"

#### 方案 A：单视角（所有角色看到相同内容）

```
运动员提交反馈 → AI 分析 → 所有人看到同一份报告
```

**问题**：
- 运动员看到"ACWR 1.8，超过安全阈值"会恐慌
- 运动员看到"情绪极性：消极，动机水平：下降"会感到被"分析"
- 运动员看到"建议降低训练强度 20%"可能自行减量，打乱教练计划
- 教练看到"今天跑的还行"这种口语化描述，缺乏专业决策信息

#### 方案 B：双视角（运动员和教练看到不同内容）

```
运动员提交反馈 → AI 分析 → 生成两份报告
  ├─ 运动员视角：训练日记（事实记录 + 鼓励 + 关怀）
  └─ 教练视角：战术分析师（专业评估 + 风险预警 + 决策建议）
```

**我们选择方案 B，理由如下**：

| 维度 | 运动员视角 | 教练视角 |
|---|---|---|
| **定位** | 训练日记本 | 战术分析师 |
| **语气** | 温暖、自然、第一人称 | 专业、客观、结构化 |
| **内容** | 今天做了什么、哪里做得好、下次注意什么 | 强度区间评估、负荷管理、周期匹配度、风险等级 |
| **不包含** | ACWR、VDOT、训练区间、任何专业术语 | 运动员的原始口语描述（已转化为专业分析） |
| **目的** | 让运动员感到被关心、被记录 | 为教练提供完整的决策支持 |

#### 信息分级的具体实现

```
┌─────────────────────────────────────────────────────┐
│                   AI 分析引擎                         │
│                                                     │
│  输入：运动员语音 + 主观评分 + FIT 数据 + 周期数据      │
│                                                     │
│  输出：                                              │
│  ┌─────────────────────┐  ┌───────────────────────┐ │
│  │    运动员视角         │  │     教练视角           │ │
│  │                     │  │                       │ │
│  │ • summary (事实描述) │  │ • detailed_analysis   │ │
│  │ • training_log      │  │ • zone_assessment     │ │
│  │ • highlights        │  │ • periodization_note  │ │
│  │ • encouragement     │  │ • psychology_analysis │ │
│  │ • care_message      │  │ • risk_assessment     │ │
│  │                     │  │ • ai_suggestion       │ │
│  │ ❌ 无专业术语        │  │ ✅ 完整专业分析        │ │
│  └─────────────────────┘  └───────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

#### care_message 的设计细节

当 AI 检测到运动员存在消极情绪时，会生成一句温暖的关怀语。这个功能的设计原则是：

- **绝不能暴露心理分析内容**：不能说"检测到你有焦虑倾向"，而要说"最近训练辛苦了，记得好好休息"
- **像朋友一样说话**：不用"请关注您的心理状态"这种官方措辞
- **积极情绪时生成鼓励**："今天状态不错，继续保持！"
- **中性情绪时不生成**：避免过度关怀造成压力

### 2.3 为什么需要"治疗-训练冲突检查"

#### 场景还原

```
周一上午：运动员小张跟队医说膝盖疼
周一下午：队医对小张做了超声波治疗，建议"3天内避免高强度冲击"
周一晚上：主教练制定周二训练计划，安排小张跑间歇（I 区，高强度）
周二上午：小张带伤跑间歇，膝盖疼痛加重
周二下午：队医发现后质问教练"为什么安排高强度"
教练回答："我不知道他昨天做了治疗"
```

#### 传统解决方式的问题

- 依赖运动员自己转述（不可靠，运动员可能隐瞒或遗忘）
- 依赖队医和教练的非正式沟通（容易遗漏）
- 依赖微信群消息（信息淹没在聊天记录中）

#### 我们的解决方案

```
队医录入治疗记录（部位：膝盖，方法：超声波，恢复期：3天）
                    ↓
系统自动映射：膝盖 → [跑步, 深蹲, 跳跃, 下肢冲击]
                    ↓
系统检索明日训练计划：小张 → 间歇跑（I 区，高强度冲击）
                    ↓
冲突检测：治疗部位(膝盖) ∩ 训练区域(下肢冲击) = 冲突
         恢复期(3天) ∩ 明日 = 仍在恢复期内
                    ↓
冲突级别：major_conflict
                    ↓
自动通知：主教练、助教
         "⚠️ 小张的膝盖治疗与明日间歇训练存在冲突，建议调整"
```

#### 设计决策

| 决策 | 选择 | 理由 |
|---|---|---|
| 冲突检测触发时机 | 队医录入治疗后自动执行 | 越早发现越好，不依赖人工触发 |
| 冲突分级 | 三级（无/轻微/严重） | 避免"狼来了"效应，只对高风险冲突推送强通知 |
| 通知对象 | 主教练 + 助教 | 确保至少一人看到并处理 |
| 身体部位映射 | 预定义映射表 | "膝盖"映射到"跑步/深蹲/跳跃"，不需要队医手动标注训练影响 |

### 2.4 为什么需要"初始能力评估"

#### 问题：没有基线数据，AI 分析就是"空中楼阁"

系统的 AI 分析引擎依赖丹尼尔斯 VDOT 体系来评估训练质量。VDOT 是什么？简单说，它是一个综合反映运动员有氧能力的数值，由比赛成绩推算而来。有了 VDOT，系统才能知道：

- 这位运动员的 E 区轻松跑配速应该是 5:30/km 而不是 6:30/km
- 今天他在 T 区跑了 4:25/km，这对他的水平来说是"正常发挥"还是"超常发挥"
- 他的训练负荷是否合理（太快/太慢/刚好）

**没有 VDOT，AI 就像一个不认识学生的老师——不知道谁是尖子谁是后进生，只能给出泛泛的评价。**

#### 传统方案的困境

| 方案 | 问题 |
|---|---|
| 让运动员自己填写 VDOT | 运动员不知道什么是 VDOT |
| 让教练手动输入 | 增加教练负担，且教练可能没有精确数据 |
| 从历史数据自动推算 | 新运动员没有历史数据 |
| 不设基线，用默认值 | 默认值对高水平运动员和初学者都偏差很大 |

#### 我们的方案：首次登录引导 + 两种录入方式

```
运动员首次登录
      ↓
系统检测到无体测数据 → 强制展示"初始能力评估"页面
      ↓
┌─────────────────┬──────────────────┐
│  方式 A: PB 成绩  │  方式 B: Conconi  │
│  (推荐，更简单)    │  (更精确)         │
│                 │                  │
│  输入:           │  输入:            │
│  · 比赛项目       │  · 有氧阈心率      │
│  · 完赛时间       │  · 有氧阈配速      │
│  · 最大心率(选填) │  · 无氧阈心率      │
│                 │  · 无氧阈配速      │
│                 │  · 最大心率        │
│                 │                  │
│  计算:           │  计算:            │
│  PB → VDOT      │  阈值 → VDOT      │
│  → 5 区训练区间   │  → 5 区训练区间    │
└─────────────────┴──────────────────┘
      ↓
  结果预览: VDOT 48.5
  E区 5:30-6:10 | M区 4:45 | T区 4:25 | I区 4:05 | R区 3:50
      ↓
  ✅ 保存 → 进入系统
```

#### 设计决策

| 决策 | 选择 | 理由 |
|---|---|---|
| 首次登录是否强制 | 是，无法跳过 | 没有基线数据，AI 分析质量无法保证 |
| 是否允许跳过 | 允许（"稍后设置"） | 降低首次使用的挫败感，但功能受限 |
| 推荐哪种方式 | PB 成绩 | 运动员大多知道自己的比赛成绩，不需要专业测试设备 |
| 更新频率 | 建议每 4-6 周 | 有氧能力变化周期约为 4-6 周，过期数据会导致区间偏差 |
| 过期提醒 | 超过 6 周首页显示红色警告 | 温和提醒，不强制 |
| 更新入口 | 首页 📊 卡片 + 全屏弹窗 | 随时可更新，不阻断正常使用 |

#### 为什么选择 VDOT 而非其他指标

| 指标 | 优势 | 劣势 |
|---|---|---|
| **VDOT（我们的选择）** | 直接关联训练配速，丹尼尔斯理论的核心参数，有大量研究验证 | 需要比赛成绩或专业测试 |
| 乳酸阈 | 生理学上更精确 | 需要实验室测试，普通运动员无法获得 |
| 最大心率 | 容易测量 | 与训练配速没有直接换算关系 |
| FTP（自行车） | 自行车领域标准 | 不适用于跑步 |

VDOT 是"用比赛成绩反推训练参数"的最佳桥梁——运动员知道自己的 10km 成绩，系统就能算出他应该怎么练。

### 2.5 为什么整合三大理论而不是只用一种

#### 问题：单一理论的局限性

| 理论 | 能回答的问题 | 不能回答的问题 |
|---|---|---|
| **丹尼尔斯** | 今天跑多快？心率控制在多少？ | 为什么最近情绪低落？该不该减量？ |
| **邦帕** | 这周该加量还是减量？赛前怎么减？ | 今天配速掉速是因为疲劳还是心理？ |
| **运动心理学** | 运动员是不是有焦虑？动机是否下降？ | 具体该怎么调整训练负荷？ |

#### 我们的方案：三理论融合

```
丹尼尔斯 → 微观执行层：配速、心率、训练量限制
    ↕
邦帕 → 宏观策略层：周期、负荷节奏、赛前减量
    ↕
运动心理学 → 心理状态层：情绪、动机、自信、焦虑
```

**AI 分析时的优先级规则**：
- 三者一致时综合引用
- 侧重点不同时分别从各自角度给出分析
- 例：丹尼尔斯说"今天 I 区训练质量良好"，但运动心理学检测到"动机下降"→ 教练端报告同时展示两个维度，并给出综合建议

---

### 2.6 为什么用邀请码而不是管理员审批

#### 问题：多支队伍共用一个系统时，如何隔离数据

当系统部署为多团队共用时，A 队的教练不能看到 B 队的运动员数据，B 队的通知不能发到 A 队。需要一种机制让用户加入正确的团队。

#### 方案对比

| 方案 | 优势 | 劣势 |
|---|---|---|
| **邀请码（我们的选择）** | 教练发码、队员粘贴，零学习成本 | 码泄露风险（可通过重置码解决） |
| 管理员审批 | 安全性最高 | 管理员需要逐一审批，队员需要等待 |
| 扫码加入 | 最流畅 | 链接可泄露，需要防刷机制 |

#### 实现机制

```
主教练注册 → 创建团队 → 系统生成 6 位邀请码 → 微信发给队员
队员注册 → 输入邀请码 → 自动关联到团队 → 只能看到本队数据
```

**数据隔离范围**：训练记录、训练计划、通知、伤病/治疗记录、绑定关系、用户列表——所有查询都自动加 `team_id` 过滤，跨队数据完全隔离。

### 2.7 语音识别的链路设计

#### 为什么链路这么长

```
浏览器录音 → WebM → 转 WAV → 转 Base64 → 发后端 → 调 MIMO ASR → 返回文字
```

每一步都有存在的理由：

| 步骤 | 为什么需要 |
|---|---|
| 浏览器录音 | 必须通过浏览器获取麦克风 |
| 转 WAV | MIMO ASR 要求 WAV 格式输入 |
| 转 Base64 | JSON 不能直接传二进制数据 |
| 发后端 | 保护 API Key 不暴露在前端 |
| 调 MIMO ASR | 专业模型，中文识别准确率高 |

#### 为什么不用浏览器自带的 Web Speech API

Web Speech API（`webkitSpeechRecognition`）可以一步到位语音转文字，但在国内有两个致命问题：

1. 依赖 Google 服务器，国内网络不稳定
2. 微信内置浏览器不支持

MIMO ASR 是小米的模型，国内服务器访问快，兼容性更好。

#### HTTPS 要求

浏览器的麦克风权限（`getUserMedia`）要求 HTTPS 环境。没有 HTTPS 时，语音功能不可用，系统自动降级为文字输入模式。

### 3.6 优先级排序逻辑

```
投入产出比 = 用户价值 × 使用频率 ÷ 实现成本

v1.0: 运动员反馈(A) + 初始评估(A) + 教练分析(A) + 冲突检查(B) → 跑通闭环

v4.0: 个性化模型(B) + 效果预测(B) + 自适应(B) → 智能化
```

**关键判断**：v1.0 的价值在于验证核心假设（运动员愿意用语音吗？双视角能被接受吗？），而非功能完整性。宁可功能少但闭环完整，也不要功能多但每个都半生不熟。

## 四、数据指标定义

### 4.1 北极星指标

**周活跃教练率**

```
定义：过去 7 天内至少登录 1 次的教练数 ÷ 总教练数
目标：≥ 80%
理由：教练是系统的核心用户，如果教练不登录，整个系统没有价值
```

选择这个指标而非"运动员活跃率"的原因：运动员的使用是被动的（教练安排了训练才有反馈可提交），教练的使用是主动的。教练活跃 = 系统在用。

## 五、关键设计决策记录

### 决策 1：为什么不用 ChatBot 形式

| 方案 | 优势 | 劣势 |
|---|---|---|
| **ChatBot（对话式）** | 交互自然 | 无法结构化输出、难以做双视角、教练需要翻聊天记录找信息 |
| **结构化表单** | 数据规范 | 输入门槛高、运动员不愿填 |
| **语音 + 结构化输出（我们选择的）** | 语音降低输入门槛，AI 解析为结构化数据，双视角输出 | 依赖 ASR 和 LLM 的准确率 |

**结论**：ChatBot 适合通用问答，但训练分析需要结构化数据做趋势分析和冲突检测。语音输入 + 结构化输出是最佳平衡点。

### 决策 2：为什么选择暗色主题

| 考量 | 说明 |
|---|---|
| **使用场景** | 运动员训练后在户外/更衣室使用，强光下暗色更易读 |
| **视觉疲劳** | 教练晚间查看大量数据，暗色减少视觉疲劳 |
| **专业感** | 暗色主题传达"专业工具"的感知，符合体育数据分析的调性 |

### 决策 3：为什么选择 JSON 文件存储而非传统数据库

| 考量 | 说明 |
|---|---|
| **部署简单** | 零外部依赖，`npm install && npm run dev` 即可运行 |
| **数据可读** | JSON 文件可直接打开查看，调试方便 |
| **适合规模** | 单支运动队（20-50 人）的数据量完全在 JSON 文件的能力范围内 |
| **迁移路径** | 当数据量增长时，可平滑迁移到 SQLite 或 PostgreSQL（schema.sql 已预留） |

### 决策 4：为什么 AI prompt 内嵌理论而非使用通用 prompt

| 方案 | 优势 | 劣势 |
|---|---|---|
| **通用 prompt** | 简单、灵活 | 输出质量不稳定、缺乏专业深度 |
| **理论内嵌 prompt（我们选择的）** | 输出有理论支撑、可解释、可追溯 | prompt 较长、需要持续优化 |

**结论**：运动训练分析不是"让 AI 自由发挥"的场景。教练需要知道"这个建议是基于什么理论"，而非"AI 说的"。内嵌丹尼尔斯、邦帕和运动心理学理论，让 AI 的每个建议都有据可依。

### 决策 5：为什么首次登录强制体测而不是自动推算

| 方案 | 优势 | 劣势 |
|---|---|---|
| **从历史数据自动推算** | 零操作成本 | 新运动员没有历史数据；历史数据质量参差不齐（GPS 漂移、心率异常） |
| **FIT 文件自动分析** | 客观、精确 | 需要上传文件、算法复杂、结果不够稳定 |
| **手动输入 + 首次强制（我们选择的）** | 数据来源明确、运动员有参与感、VDOT 可解释 | 增加首次使用的操作步骤 |

**结论**：VDOT 是训练区间的基础参数，必须准确。让运动员输入自己最熟悉的数据（比赛成绩），系统负责转换为专业参数，是"准确性"和"易用性"的最佳平衡。首次登录强制执行是确保数据完整性的必要手段——允许跳过但限制功能，既不强迫又有引导。

### 决策 6：为什么用 AudioContext 而不是 MediaRecorder

| 方案 | 优势 | 劣势 |
|---|---|---|
| **MediaRecorder** | API 简洁，浏览器原生支持 | 微信/国产浏览器不支持 |
| **AudioContext 手动录音（我们选择的）** | 兼容性好，微信/小米浏览器都支持 | 代码复杂，需要手动处理音频缓冲区 |
| **Web Speech API** | 零后端，一步到位 | 依赖 Google 服务器，国内不稳定 |

**结论**：运动员训练完大概率用微信扫码打开系统，必须兼容微信浏览器。AudioContext 是唯一在微信中可用的录音方案。系统自动检测浏览器能力，优先用 MediaRecorder，不支持时降级到 AudioContext。

## 六、风险与应对

| 风险 | 可能性 | 影响 | 应对策略 |
|---|---|---|---|
| 运动员不愿意用语音（觉得尴尬） | 中 | 高 | 提供文字备选；在更衣室等私密场景引导使用 |
| 语音功能需要 HTTPS 环境 | 高 | 高 | 无 HTTPS 时自动降级为文字输入；长期方案配域名 + SSL 证书 |
| 运动员不知道自己的比赛成绩 | 低 | 中 | 提供 Conconi 测试备选；允许跳过但提示功能受限 |
