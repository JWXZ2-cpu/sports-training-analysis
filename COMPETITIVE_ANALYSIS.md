# Sigma 产品解剖（Product Teardown）

> 深度竞品分析：Sigma vs MAI
> 分析框架：产品战略 → 用户画像 → 增长飞轮 → AI 定位 → 信息架构 → 商业模式 → 产品边界 → MAI 的机会
> 最后更新：2026-06-30

---

## 一、一句话定义

**Sigma 是一个以 AI 为核心、围绕「个人跑者」构建的跑步记录与陪伴产品。**

注意三个关键词：

- **AI 为核心**：不是"有 AI 功能"，而是"AI 是产品体验的底层"
- **个人跑者**：只有一个用户角色，不是团队
- **陪伴**：卖的不是训练系统，是情绪价值

这决定了 Sigma 的一切产品决策。

---

## 二、产品战略：Why Sigma Exists

### 2.1 Sigma 的产品假设

> 一个跑者，只要有数据，就能获得更好的训练建议。

所以 Sigma 重点做的是：
- 数据记录（GPS、手表、跑步轨迹）
- AI 分析（配速、心率、趋势）
- AI 训练计划（个性化建议）
- 跑步体验（地图日记、跑步人格、分享卡片）
- 仪式感（跑后长图、回忆、跑量赛）

### 2.2 Sigma 真正卖的是什么

Sigma 官网和 App Store 强调的关键词：

```
真正懂你 · 仪式感 · AI 伙伴 · 回忆 · 地图日记 · 跑步人格 · AI 训练计划
```

而不是：

```
训练负荷 · 乳酸阈 · 训练周期 · 伤病预测 · ACWR
```

**说明：Sigma 真正卖的是陪伴感，不是训练系统。**

### 2.3 产品战略：AI First, Emotion Driven

Sigma 的更新日志几乎都是：
- 地图日记
- 跑步人格
- 世界杯跑量赛
- 校园跑量赛
- 分享卡片
- 跑后长图
- Sig 聊天
- 陌生名词解释
- 跑步 Tips

很少出现：
- VO2max 预测
- 疲劳模型
- 训练周期优化
- 恢复指数

**结论：AI 只是手段，情绪价值才是产品。**

## 三、用户画像与核心场景

### 3.1 Sigma 的用户画像

| 维度 | 描述 |
|---|---|
| **身份** | 业余跑者、跑步爱好者 |
| **年龄** | 20-40 岁 |
| **跑步频率** | 每周 2-5 次 |
| **核心诉求** | 记录跑步、获得反馈、坚持下去 |
| **痛点** | 跑步孤独、缺乏动力、不知道练得对不对 |
| **使用场景** | 跑步前后、跑步中、社交分享 |

### 3.2 核心场景

**场景一：跑后记录**
```
跑完步 → 打开 Sigma → 自动记录轨迹 → AI 生成分析 → 看到鼓励 → 分享到朋友圈
```

**场景二：训练计划**
```
想提升成绩 → 告诉 Sigma 目标 → AI 生成计划 → 按计划执行 → AI 调整
```

**场景三：社交激励**
```
看到朋友在跑 → 加入跑量赛 → 互相点赞 → 坚持跑步
```

### 3.3 MAI 的用户画像对比

| 维度 | Sigma | MAI |
|---|---|---|
| **身份** | 业余跑者 | 专业运动队成员 |
| **角色** | 1 个（跑者） | 5 个（运动员、教练、助教、队医、管理） |
| **核心诉求** | 坚持跑步、获得陪伴 | 提升训练质量、团队协同 |
| **痛点** | 孤独、缺乏动力 | 信息不对称、决策缺乏数据支撑 |
| **使用场景** | 个人跑步 | 团队训练管理 |

## 四、北极星指标与增长飞轮

### 4.1 Sigma 的北极星（推测）

**周活跃跑者数**

理由：
- Sigma 的核心价值是帮助用户"坚持跑步"
- 用户只要在跑，就会用 Sigma 记录
- 周活跃跑者数直接反映产品核心价值的交付

### 4.2 Sigma 的增长飞轮

```
跑步
 ↓
记录（GPS、轨迹、配速、心率）
 ↓
AI 分析（个性化报告、跑步人格）
 ↓
分享（地图日记、跑后长图、跑量赛）
 ↓
获得反馈（点赞、评论、排名）
 ↓
继续跑（动力增强）
 ↓
更多数据
 ↓
AI 越来越懂你（个性化程度提升）
 ↓
体验越来越好
 ↓
继续跑
```

**关键洞察**：Sigma 的飞轮核心驱动力是**社交反馈 + 情感连接**，不是训练效果。用户不是因为"跑得更快了"而继续用，而是因为"跑完有好看的分享卡片"而继续用。

### 4.3 MAI 的增长飞轮（对比）

```
运动员反馈（语音/文字）
 ↓
AI 分析（双视角报告）
 ↓
教练调整训练计划
 ↓
训练效果更好
 ↓
运动员更愿意反馈
 ↓
更多数据
 ↓
AI 分析更准确
 ↓
教练更依赖系统
 ↓
团队训练效率提升
```

**关键区别**：
- Sigma 的飞轮靠**情感驱动**（分享 → 社交反馈 → 动力）
- MAI 的飞轮靠**效果驱动**（反馈 → 决策 → 效果 → 信任）

## 五、AI 在产品中的角色定位

### 5.1 Sigma 的 AI 角色

**AI 是陪伴者，不是教练。**

| 功能 | AI 的角色 | 用户感知 |
|---|---|---|
| 跑步分析 | "你今天跑得不错" | 被认可 |
| 跑步人格 | "你是稳健型跑者" | 被理解 |
| AI 训练计划 | "建议明天轻松跑" | 被指导 |
| Sig 聊天 | "最近跑步怎么样？" | 被关心 |

**Sigma 的 AI 不做**：
- 不做负荷管理
- 不做伤病预测
- 不做周期化分析
- 不做跨角色信息分发

### 5.2 MAI 的 AI 角色

**AI 是决策助手，不是伙伴。**

| 功能 | AI 的角色 | 用户感知 |
|---|---|---|
| 训练分析 | "I 区训练质量良好，但动机下降" | 获得决策依据 |
| 双视角报告 | 运动员看日记，教练看分析 | 信息分级 |
| 冲突检测 | "膝盖治疗与明日间歇训练冲突" | 风险预警 |
| 计划生成 | "基于 VDOT 和周期，生成本周计划" | 辅助决策 |

**MAI 的 AI 不做**：
- 不做社交陪伴
- 不做情绪价值
- 不做分享卡片
- 不做跑步人格

### 5.3 本质区别

| 维度 | Sigma | MAI |
|---|---|---|
| **AI 的目标** | 让用户继续跑 | 让教练做出更好的决策 |
| **AI 的输出** | 鼓励、陪伴、个性化 | 分析、预警、建议 |
| **AI 的衡量** | 用户留存、使用时长 | 教练采纳率、训练效果 |
| **AI 的学习** | 越跑越懂你 | 越分析越准确 |

## 六、信息架构与核心交互路径

### 6.1 Sigma 的信息架构（推测）

```
首页
├── 跑步记录（GPS、轨迹、配速、心率）
├── AI 分析（本次跑步报告）
├── 训练计划（AI 生成的周计划）
├── 跑步人格（性格分析）
├── 地图日记（可视化轨迹）
├── 社区（跑量赛、排行榜）
└── 个人中心
    ├── 历史记录
    ├── 数据统计
    └── 设置
```

**特点**：
- 单一用户角色，不需要权限管理
- 信息平面化，所有功能对同一用户可见
- 社交功能深度整合

### 6.2 MAI 的信息架构

```
登录 → 角色路由
├── 运动员端
│   ├── 首页（计划 + 反馈入口）
│   ├── 训练日记
│   ├── 报告详情（简化版）
│   ├── 数据汇总
│   ├── 体测评估
│   └── 个人资料
├── 主教练端
│   ├── 工作台（团队总览）
│   ├── 计划管理（周视图）
│   ├── AI 辅助计划
│   ├── 训练课记录
│   ├── 团队管理（邀请码/成员）
│   └── 通知
├── 助教端
│   ├── 工作台（运动员列表）
│   ├── 训练备注
│   └── 报告翻译
├── 队医端
│   ├── 伤病管理
│   ├── 治疗记录
│   └── 冲突检查
└── 管理端
    ├── 考勤统计
    └── 团队状态
```

**特点**：
- 5 个角色，信息严格隔离
- 每个角色看到的数据和功能完全不同
- 需要 RBAC 权限系统

### 6.3 信息架构的本质区别

| 维度 | Sigma | MAI |
|---|---|---|
| **用户角色** | 1 个 | 5 个 |
| **信息流向** | 单向（AI → 用户） | 多向（运动员 ↔ 教练 ↔ 队医） |
| **权限模型** | 无 | RBAC 角色权限 |
| **数据隔离** | 无（个人数据） | 有（团队隔离） |

## 七、商业模式与竞争壁垒

### 7.1 Sigma 的商业模式（推测）

| 收入来源 | 说明 |
|---|---|
| **订阅制** | 高级 AI 功能、训练计划、数据分析 |
| **增值服务** | 个性化训练计划、赛事报名 |
| **广告/合作** | 运动品牌合作、赛事赞助 |
| **电商** | 跑步装备推荐 |

### 7.2 Sigma 的竞争壁垒

**壁垒一：长期跑步历史数据**

不是 GPS 数据（谁都有），而是：
- 长期训练记录
- 配速变化趋势
- 心率变化趋势
- 跑步习惯数据

"越跑越聪明"就是壁垒。

**壁垒二：情感设计**

- 地图日记
- 跑步人格
- 跑量赛
- 分享卡片

这些不是功能，是 Retention Design（留存设计）。

**壁垒三：设计感**

Sigma 的视觉特别"Apple"：
- 克制、简洁
- 没有大量按钮
- 没有很多菜单
- 设计本身就是竞争力

### 7.3 MAI 的竞争壁垒（对比）

| 壁垒 | Sigma | MAI |
|---|---|---|
| **数据** | 个人跑步历史 | 多角色协同数据（反馈 + 医疗 + 心理） |
| **网络效应** | 社交网络（跑友圈） | 组织网络（教练-运动员-队医） |
| **切换成本** | 丢失历史数据 | 团队迁移成本高 |
| **设计** | 情感化设计 | 专业级信息密度 |

## 八、产品边界与未覆盖需求

### 8.1 Sigma 的产品边界

**边界一：只有一个用户**

就是跑者。没有教练、没有队医、没有团队。

**边界二：AI 只能建议，不能参与训练流程**

- 可以说"建议明天轻松跑"
- 但不会修改训练计划
- 不会通知教练
- 不会同步队医

**边界三：缺少组织协同**

整个产品没有：
- RBAC 权限
- 团队管理
- 审批流程
- 训练计划流转
- 信息分级

### 8.2 Sigma 未覆盖的需求

| 需求 | Sigma 是否满足 | 说明 |
|---|---|---|
| 教练需要看到运动员真实状态 | ❌ | 没有教练角色 |
| 队医需要知道训练安排 | ❌ | 没有队医角色 |
| 治疗与训练的冲突检测 | ❌ | 没有医疗数据 |
| 运动员心理状态分析 | ❌ | 没有心理评估 |
| 多角色信息分级 | ❌ | 单一用户视角 |
| 团队训练计划管理 | ❌ | 只有个人计划 |
| 训练周期化管理 | ❌ | 没有周期理论 |

### 8.3 为什么 Sigma 不做这些

不是做不了，而是**产品定位不允许**。

Sigma 的核心假设是：
> 一个跑者，只要有数据，就能获得更好的训练建议。

如果 Sigma 加入了教练、队医、团队管理，它就不再是"陪伴型跑步产品"，而是"团队训练管理系统"。

这是两个完全不同的产品。

## 九、MAI 的切入机会：Why MAI

### 9.1 市场空白

横轴：个人 → 团队
纵轴：记录 → 决策

```
            决策
              ▲
              │
         MAI  │  （团队 + 决策）
              │
              │
记录 ─────────┼─────────────► 团队
              │
     Sigma    │  （个人 + 记录）
              │
              │
            个人
```

**Sigma 占据了右下角（个人 + 记录），MAI 占据了左上角（团队 + 决策）。**

右上角（团队 + 记录）是 TrainingPeaks 等传统训练管理工具。
左下角（个人 + 决策）是各种 AI 跑步教练 App。

### 9.2 为什么专业运动队还需要 MAI

**Sigma 已经证明了：AI + 跑步记录是有价值的。**

但它主要服务于个人跑者（C 端）。

专业运动队真正缺少的是：

| 缺失 | Sigma 的现状 | MAI 的解决方案 |
|---|---|---|
| 教练不知道运动员真实状态 | 没有教练角色 | 双视角 AI 报告 |
| 队医不知道训练安排 | 没有队医角色 | 医训冲突检测 |
| 助教不知道恢复情况 | 没有助教角色 | 全队数据查看 |
| 管理者不知道整体状态 | 没有管理角色 | 考勤/状态看板 |
| 信息在口头对话中丢失 | 个人记录 | 语音结构化反馈 |

### 9.3 MAI 的产品假设（对比 Sigma）

**Sigma 的假设**：
> 一个跑者，只要有数据，就能获得更好的训练建议。

**MAI 的假设**：
> 在专业运动队里，真正的问题不是"不会跑"，而是"信息无法流动"。

因此，MAI 解决的是：
- 教练不知道运动员真实状态
- 队医不知道训练安排
- 助教不知道恢复情况
- 管理者不知道整体状态

**AI 只是中间的连接器。**

### 9.4 一句话概括

> Sigma 解决的是「一个人如何长期坚持跑步」；MAI 解决的是「一支专业运动队如何基于 AI 做出更科学的训练决策」。

**这是产品定位的本质区别，而不是功能多少的区别。**

## 十一、对 MAI 产品设计的启示

### 11.1 可以学习 Sigma 的地方

| 方面 | Sigma 的做法 | MAI 可以借鉴 |
|---|---|---|
| **情感化设计** | 地图日记、跑步人格、分享卡片 | 运动员端的训练日记可以增加情感化元素 |
| **仪式感** | 跑后长图、回忆功能 | 训练完成后的成就感展示 |
| **设计克制** | 少按钮、简洁界面 | 保持当前的暗色主题 + 极简设计 |
| **数据飞轮** | 越跑越懂你 | 越分析越准确 |

### 11.2 不应该学习 Sigma 的地方

| 方面 | Sigma 的做法 | MAI 不应该做 |
|---|---|---|
| **社交功能** | 跑友圈、排行榜 | 运动员之间的社交会分散注意力 |
| **情绪价值优先** | 陪伴 > 分析 | 专业运动队需要的是决策支持，不是陪伴 |
| **单用户设计** | 只有跑者 | 必须保持多角色协同 |
| **C 端增长策略** | 社交裂变、分享卡片 | B 端应该靠教练口碑传播 |

### 11.3 MAI 的差异化定位

```
Sigma: AI + 跑步记录 + 情感陪伴 → 个人跑者坚持跑步
MAI:   AI + 多角色协同 + 决策支持 → 专业运动队提升训练质量
```

**不是要做一个比 Sigma 更好的跑步 App，而是解决另一类用户尚未被满足的需求。**

## 十二、面试表达建议

### 1 分钟版本

"我分析了 Sigma，它是一个非常优秀的 AI 跑步陪伴产品。它的核心假设是'一个跑者只要有数据就能获得更好的建议'，所以它重点做的是记录、AI 分析、情感设计和社交激励。

但 Sigma 的产品边界很清晰：只有一个用户角色（跑者），AI 只能建议不能参与训练流程，缺少组织协同。

而我做的 MAI 解决的是另一类问题：在专业运动队里，真正的问题不是'不会跑'，而是'信息无法流动'——教练不知道运动员真实状态，队医不知道训练安排。所以 MAI 做的是多角色协同和 AI 决策支持。

这不是功能多少的区别，是产品定位的本质区别。"

### 3 分钟版本（面试时用）

用以下顺序讲：

1. **Sigma 是什么**（一句话定义 + 产品假设）
2. **Sigma 做对了什么**（AI First + Emotion Driven + 数据飞轮）
3. **Sigma 的产品边界**（单用户、AI 只能建议、缺少协同）
4. **MAI 的机会**（团队 + 决策象限的空白）
5. **为什么不是同一个产品**（用户不同、问题不同、AI 角色不同）
